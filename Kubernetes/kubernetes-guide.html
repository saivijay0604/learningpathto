What is Kubernetes?

Start here — understand the problem K8s solves before diving into how it works.

🐳

The Problem: Running Many Containers

Imagine you have a web app split into 10 microservices, each running as a Docker container. You need them to start automatically, restart if they crash, scale up when traffic spikes, and communicate with each other. Doing this manually is chaos.


Kubernetes is an open-source container orchestration platform that automates all of this — deploying, scaling, healing, and managing containers across a cluster of machines.

✅

What K8s Does

🔁 Auto-restarts crashed containers
📈 Scales up/down based on load
🔄 Rolling updates with zero downtime
🌐 Load balancing across pods
🔐 Secret & config management
💾 Persistent storage management

🔑

Key Terms to Know

Cluster — group of machines running K8s
Node — a single machine in the cluster
Pod — smallest deployable unit (1+ containers)
Service — stable network endpoint for pods
Namespace — virtual cluster within a cluster
kubectl — CLI tool to control K8s

📖

K8s vs Docker — What's the Difference?

Docker builds and runs individual containers. Kubernetes manages and orchestrates hundreds of containers across multiple machines. They work together — Docker (or containerd) is the container runtime, and Kubernetes is the orchestrator on top.

# Docker: run ONE container
docker run -p 80:80 nginx

# Kubernetes: declare desired state for MANY containers across machines
kubectl apply -f nginx-deployment.yaml

Control Plane Components

The brain of the cluster. It makes global decisions and manages the cluster state.


COMPONENTROLE
kube-apiserverThe front door — all commands go through this REST API
etcdKey-value store that holds ALL cluster state/config
kube-schedulerDecides which node each new pod runs on
kube-controller-managerRuns controllers that maintain desired state (e.g. ReplicaSet)
cloud-controller-managerIntegrates with cloud providers (AWS, GCP, Azure)
💪

Worker Node Components

Every worker node runs these three components to host your application pods.


COMPONENTROLE
kubeletAgent that ensures containers are running in pods as specified
kube-proxyHandles network routing and load balancing between pods
container runtimeActually runs containers — Docker, containerd, or CRI-O
⚡

How It All Works Together

# 1. You run a command
kubectl apply -f deployment.yaml

# 2. kubectl sends it to kube-apiserver
# 3. apiserver saves it to etcd
# 4. kube-scheduler picks a node for the pod
# 5. kubelet on that node pulls the image & starts the container
# 6. kube-proxy sets up networking
# 7. Your app is running! 🎉
Core Kubernetes Objects

These are the building blocks you'll use every day. Each is a declarative API object.

📦

Pod Most Basic Unit

A Pod is the smallest deployable unit. It wraps one or more containers that share network and storage. Pods are ephemeral — if one dies, a new one is created with a different IP.

apiVersion: v1
kind: Pod
metadata:
  name: my-pod
spec:
  containers:
  - name: nginx
    image: nginx:latest
    ports:
    - containerPort: 80
🔁

Deployment Most Common

Manages a ReplicaSet of identical Pods. Handles rolling updates, rollbacks, and scaling. This is what you use to run your app in production.

apiVersion: apps/v1
kind: Deployment
metadata:
  name: web-app
spec:
  replicas: 3          # Run 3 copies
  selector:
    matchLabels:
      app: web
  template:
    metadata:
      labels:
        app: web
    spec:
      containers:
      - name: web
        image: myapp:v2
🌐

Service

Gives pods a stable IP and DNS name. Pods come and go, but the Service stays put. Types:


ClusterIP — internal only (default)
NodePort — expose on each node's IP
LoadBalancer — cloud load balancer
ExternalName — DNS alias

🗂️

ConfigMap & Secret

ConfigMap — stores non-sensitive config (env vars, config files)


Secret — stores sensitive data (passwords, API keys) base64-encoded. Use with external secret managers in prod!

💾

PersistentVolume (PV/PVC)

Pods are stateless by default. PersistentVolume provides durable storage. PVC (Claim) is how a pod requests storage. Think of PV as a disk, PVC as a reservation for that disk.

🔁

StatefulSet

Like a Deployment but for stateful apps (databases, Kafka). Pods get stable names (pod-0, pod-1) and persistent storage that follows them even on restarts.

kubectl is your CLI to communicate with the Kubernetes API. Master these commands.

Setup & Context

# Check version
kubectl version

# View cluster info
kubectl cluster-info

# List all contexts (clusters)
kubectl config get-contexts

# Switch context
kubectl config use-context my-cluster

# Set default namespace
kubectl config set-context --current --namespace=dev
🔍

Viewing Resources

# List all pods
kubectl get pods
kubectl get pods -A                 # All namespaces
kubectl get pods -o wide            # Show node info

# Describe a resource (full details)
kubectl describe pod my-pod
kubectl describe node worker-1

# View logs
kubectl logs my-pod
kubectl logs my-pod -f            # Follow/stream logs
kubectl logs my-pod -c container1  # Specific container

# Get all resources
kubectl get all
🚀

Creating & Managing

# Apply a YAML file (create or update)
kubectl apply -f deployment.yaml
kubectl apply -f ./k8s/           # Apply all files in dir

# Delete resources
kubectl delete -f deployment.yaml
kubectl delete pod my-pod

# Scale a deployment
kubectl scale deployment web-app --replicas=5

# Rolling update (new image)
kubectl set image deployment/web-app web=myapp:v3

# Rollback
kubectl rollout undo deployment/web-app

# Check rollout status
kubectl rollout status deployment/web-app
🔧

Debugging

# Shell into a running pod
kubectl exec -it my-pod -- /bin/bash

# Port-forward to access locally
kubectl port-forward pod/my-pod 8080:80
kubectl port-forward svc/my-service 8080:80

# Run a temporary debug pod
kubectl run debug --rm -it --image=busybox -- sh

# Watch pods in real-time
kubectl get pods -w

# Get events (great for debugging)
kubectl get events --sort-by='.lastTimestamp'
Writing YAML Manifests

Kubernetes resources are defined as YAML files. Here are real-world examples.

📄

Complete App Deployment

A typical setup has a Deployment + Service. Here's a full working example:

--- # deployment.yaml
apiVersion: apps/v1
kind: Deployment
metadata:
  name: web-app
  namespace: production
  labels:
    app: web
    version: v2
spec:
  replicas: 3
  selector:
    matchLabels:
      app: web
  strategy:
    type: RollingUpdate
    rollingUpdate:
      maxSurge: 1
      maxUnavailable: 0
  template:
    metadata:
      labels:
        app: web
    spec:
      containers:
      - name: web
        image: myapp:v2
        ports:
        - containerPort: 3000
        env:
        - name: NODE_ENV
          value: production
        - name: DB_PASSWORD
          valueFrom:
            secretKeyRef:
              name: db-secret
              key: password
        resources:
          requests:
            cpu: 100m
            memory: 128Mi
          limits:
            cpu: 500m
            memory: 512Mi
        livenessProbe:
          httpGet:
            path: /health
            port: 3000
          initialDelaySeconds: 10
        readinessProbe:
          httpGet:
            path: /ready
            port: 3000
---
apiVersion: v1
kind: Service
metadata:
  name: web-app-svc
spec:
  selector:
    app: web
  ports:
  - port: 80
    targetPort: 3000
  type: LoadBalancer

Networking in Kubernetes

How pods talk to each other and the outside world.

🌐

Networking Rules

K8s networking follows three key rules:


1️⃣ Every pod gets its own IP — no NAT between pods
2️⃣ All pods can talk to all other pods without port mapping
3️⃣ Nodes can talk to pods and vice versa

🔀

Service Types

ClusterIP
10.96.0.1:80 — internal only, default


NodePort
nodeIP:30080 — accessible from outside on each node


LoadBalancer
Cloud provider creates an external LB with a public IP

🚪

Ingress

An Ingress routes HTTP/HTTPS traffic to Services based on domain or path. Think of it as a reverse proxy / API gateway for your cluster.


# Route traffic:
api.example.com → api-svc
app.example.com → frontend-svc
/v1/users       → user-svc
📡

DNS in Kubernetes

K8s has built-in DNS (CoreDNS). Services get DNS names automatically:

# Format: service-name.namespace.svc.cluster.local
my-service.default.svc.cluster.local

# Within same namespace, just use service name:
curl http://web-app-svc/api/users

# Cross-namespace:
curl http://web-app-svc.production.svc.cluster.local
🔒

NetworkPolicy

By default, all pods can talk to all pods. NetworkPolicy lets you restrict traffic — like a firewall for pod-to-pod communication.

# Only allow traffic from pods with label app=frontend
spec:
  podSelector:
    matchLabels:
      app: database
  ingress:
  - from:
    - podSelector:
        matchLabels:
          app: frontend
Helm Charts ⛵

Helm is the package manager for Kubernetes — like npm for Node.js or apt for Ubuntu.

⛵

What is Helm?

Managing dozens of YAML files for every app is painful. Helm bundles all K8s resources into a single Chart — a reusable, versioned package. You can install, upgrade, roll back, and share apps with a single command.


🔑 Chart — a Helm package (collection of YAML templates)
🔑 Release — an installed instance of a chart
🔑 Repository — a place to store and share charts
🔑 values.yaml — config file to customize a chart

📁

Helm Chart Structure

my-app/
├── Chart.yaml          # Chart metadata (name, version, description)
├── values.yaml         # Default configuration values
├── charts/             # Sub-charts (dependencies)
└── templates/          # K8s YAML templates
    ├── deployment.yaml
    ├── service.yaml
    ├── ingress.yaml
    ├── configmap.yaml
    └── _helpers.tpl    # Reusable template snippets
⚡

Essential Helm Commands

# Install Helm (macOS)
brew install helm

# Add a chart repository
helm repo add bitnami https://charts.bitnami.com/bitnami
helm repo update

# Search for charts
helm search repo nginx
helm search hub wordpress        # Search Artifact Hub

# Install a chart (creates a "release")
helm install my-nginx bitnami/nginx
helm install my-app ./my-app/    # From local chart

# Install with custom values
helm install my-app bitnami/nginx --values custom-values.yaml
helm install my-app bitnami/nginx --set replicaCount=3

# List installed releases
helm list
helm list -A                     # All namespaces

# Upgrade a release
helm upgrade my-nginx bitnami/nginx --set replicaCount=5

# Rollback to previous version
helm rollback my-nginx 1         # Roll back to revision 1

# Uninstall a release
helm uninstall my-nginx

# Show release history
helm history my-nginx
📝

Chart.yaml — Chart Metadata

apiVersion: v2
name: my-app
description: A Helm chart for my web application
type: application
version: 0.1.0          # Chart version
appVersion: "2.1.0"    # App version inside
dependencies:
- name: postgresql
  version: 12.x.x
  repository: https://charts.bitnami.com/bitnami
⚙️

values.yaml — Customization

This is the heart of Helm — define defaults that users can override.

# values.yaml
replicaCount: 2

image:
  repository: myapp
  tag: "latest"
  pullPolicy: IfNotPresent

service:
  type: ClusterIP
  port: 80

ingress:
  enabled: false
  host: myapp.example.com

resources:
  limits:
    cpu: 500m
    memory: 512Mi
  requests:
    cpu: 100m
    memory: 128Mi
🧩

Templates — Using Values in YAML

Templates use Go templating syntax {{ }} to inject values dynamically.

# templates/deployment.yaml
apiVersion: apps/v1
kind: Deployment
metadata:
  name: {{ .Release.Name }}-app
  labels:
    app: {{ .Chart.Name }}
    version: {{ .Chart.Version }}
spec:
  replicas: {{ .Values.replicaCount }}
  template:
    spec:
      containers:
      - name: app
        image: "{{ .Values.image.repository }}:{{ .Values.image.tag }}"
        {{- if .Values.ingress.enabled }}
        # This block only renders if ingress.enabled = true
        {{- end }}
